## 2. 第一轮验收：架构审计与核心逻辑验证 (v1.0)

### 2.1 核心缺陷：虚假的“解析”引擎 (P0)
*   **问题描述**: 设计文档承诺使用 ANTLR 4 进行语法解析，但实测发现 `ShellCommandExtractor.java` 仅使用了 `[;|&]` 正则硬分割。
*   **安全影响**: 引号嵌套（如 `echo "Hello; rm -rf /"`）会被切分为两条指令，导致正常逻辑被拦截（误报）或恶意指令被隐匿（绕过）。
*   **建议**: 必须实现真正的 ANTLR 解析，提取 `SimpleCommand` 节点。

### 2.2 逻辑漏洞：白名单绕过 (P0)
*   **问题描述**: 验证用例 `testSubcommandWhitelist` 失败。`Rule.matches` 使用了 `find()` 局部匹配。
*   **复现**: `ls; rm -rf /` 因为匹配了白名单 `^ls\b` 而被整串放行。
*   **建议**: 白名单匹配必须是**全量匹配**，且必须确保解析后的**所有指令片段**都安全。

### 2.3 工程化缺陷
*   **Builder 配置覆盖**: `withThreshold()` 会重置整个 `config` 对象，导致其他设置丢失。
*   **规则冲突检测缺失**: 用户在黑白名单配置冲突规则时，系统无任何提示。

## 3. 第二轮验收：深层防御与故障策略审计 (v2.0)

### 3.1 阶段一整改验证 (Progress)
*   ✅ **解析能力**: 引入了支持引号嵌套与转义的状态机提取器。
*   ✅ **工程化**: 修复了 Builder Bug，新增了 `RuleConflictChecker` 静态扫描机制。

### 3.2 致命漏洞：子 Shell 注入绕过 (P0)
*   **问题描述**: 测试 `testSubshellDetection` 证实，系统无法拦截 `echo $(rm -rf /)`。
*   **原因分析**: 
    1.  `RuleMatcher` 缺失对子 Shell 引导符（`$`, `(`, `` ` ``）的拦截。
    2.  `ANTLR` 模式下的语法定义过于简陋（`WORD+`），无法识别嵌套结构。
*   **建议**: 扩展语法定义以支持“命令替换”识别，并递归提取子指令。

### 3.3 安全策略：故障放行风险 (P0)
*   **问题描述**: 在解析异常（如输入 `!!!`）时，系统默认选择“放行”。
*   **位置**: `DetectionEngine.java` 捕获异常后默认 `passed: true`。
*   **影响**: 攻击者可通过畸形输入诱发解析失败，实现恶意指令的静默放行。
*   **整改要求**: 改为 **Fail-Safe** 模式，解析失败必须标记为 `DANGER` 拦截。
